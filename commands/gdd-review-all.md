전체 GDD 교차 검토를 시작합니다.

**워크플로우:**

1. **GDD 파일 탐색:**
   - `design/gdd/` 디렉터리의 모든 `.md` 파일 나열
   - 파일이 없으면: "GDD가 없습니다. `/gdd-concept`으로 게임 컨셉을 먼저 작성하고, `/gdd-design [시스템명]`으로 첫 번째 GDD를 작성하세요." 출력 후 종료

2. **컨텍스트 로드:**
   - `design/concept/game-concept.md` 읽기
   - `design/registry/entities.yaml` 읽기
   - 모든 `design/gdd/*.md` 파일 읽기

3. **Phase 1 — 일관성 검토:**

   다음을 찾습니다:
   - **규칙 모순**: GDD-A가 "최소값은 N"이라고 하고 GDD-B가 그것을 우회하는 경우
   - **공식 불일치**: 두 시스템이 같은 값을 다르게 정의하는 경우
   - **의존성 갭**: GDD-A가 존재하지 않는 GDD-B를 참조하는 경우
   - **소유권 충돌**: 두 GDD가 같은 값의 소유권을 주장하는 경우
   - **오래된 참조**: 다른 GDD에서 더 이상 일치하지 않는 참조

4. **Phase 2 — 설계 이론 검토:**

   다음을 찾습니다:
   - **필라 드리프트**: 어떤 필라도 서비스하지 않는 시스템
   - **지배 전략**: 항상 최적인 선택을 만드는 공식 조합
   - **인지 과부하**: 플레이어가 동시에 추적해야 하는 시스템이 너무 많음
   - **경제 불균형**: 파우셋이 싱크를 압도하거나 그 반대인 경우
   - **루도내러티브 불협화음**: 게임플레이가 내러티브 테마와 충돌

5. **발견 사항 분류:**

   각 발견 사항에 대해:
   - 심각도: HIGH (블로킹) / MEDIUM (권고) / LOW (참고)
   - 관련 파일 명시
   - 충돌하는 규칙/값 인용
   - 해결 제안

6. **리포트 출력:**

   ```
   ## 전체 GDD 교차 검토 리포트
   검토 대상: [N]개 GDD 파일
   
   ### Phase 1: 일관성
   [발견 사항 표 또는 "충돌 없음"]
   
   ### Phase 2: 설계 이론
   [발견 사항 표 또는 "문제 없음"]
   
   ### 의존성 갭
   [누락된 GDD 목록 또는 "모든 의존성 충족"]
   
   ### 권고 사항
   [구체적인 다음 단계]
   
   ---
   판정: CONSISTENT / MINOR ISSUES / MAJOR ISSUES
   ```

**판정 기준:**
- **CONSISTENT**: 블로킹 충돌 없음, 모든 의존성 충족
- **MINOR ISSUES**: 권고 수준 문제만 있음, 진행 가능
- **MAJOR ISSUES**: 블로킹 충돌 존재, 해결 후 재검토 필요

**노트:** 이 커맨드는 읽기 전용입니다. 발견된 문제 수정은 해당 GDD를 `/gdd-design`으로 열어 업데이트하세요.
